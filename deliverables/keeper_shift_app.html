KeeperPAM — Technical Details
Architecture, configuration, and integration specifics for the live target
Authentication

OAuth2 client credentials via Keeper PAM API. Client ID and secret in env vars — never in config files.

Env vars: KEEPER_CLIENT_ID, KEEPER_CLIENT_SECRET

Token refreshed proactively 60s before expiry.

SAML SSO + enforced MFA supported via Keeper Identity.

Encryption

Zero-knowledge AES-256-GCM — Keeper server never sees plaintext. Encryption/decryption client-side.

FedRAMP High authorized March 2026.

Even a full server compromise exposes only ciphertext.

3-Tier Data Hierarchy (Structural Delta from CyberArk)

Keeper uses a 3-tier model. A CyberArk Account maps to a pamUser linked to a pamMachine or pamDatabase. The cyberark-import tool handles ~20% of work; the remaining 80% is post-import transform to build correct linkage. Each CyberArk Safe → 2 Keeper Shared Folders.

PAM CONFIG
Top-level env config — defines gateway assignment, rotation policies, compliance scope. Maps from CyberArk Safe policy settings.
▼
PAM RESOURCE (pamMachine / pamDatabase)
The target system. Maps from CyberArk address + platformId. One Resource has many Users.
▼
PAM USER (pamUser)
The privileged credential. Maps from CyberArk userName + secret. Retrieved password written here and zeroed from memory immediately after import.

KSM — Replaces CCP/AAM

Keeper Secrets Manager (KSM) — 40+ native integrations replace CCP/AAM for application credential retrieval.

GitHub ActionsTerraformAnsibleKubernetesJenkinsAWS Secrets ManagerAzure Key VaultHashiCorp VaultPython SDKJava SDKC# .NET SDKGo SDK

Agent 06 scans all application code for CCP/AAM patterns and generates KSM SDK replacement packages. Full re-architecture for any app using AAM agent injection model — feature-flag toggle for safe rollback.

Keeper Gateway

Docker container (min 4 CPU / 16 GB RAM). HA cluster required at 250K+ accounts.

Handles rotation for ~25–30 platforms natively.

Network: Outbound HTTPS to Keeper cloud only. No inbound rules required.

Custom scripts needed for network gear (IOS/NX-OS/ASA) — validate before Wave 3.

KCM — Session Recording

Keeper Connection Manager — Apache Guacamole-based.

Browser-native session recording — no Windows RDS licensing required.

Replaces CyberArk PSM. PSM session history cannot be migrated — archive before P7 decommission.

Configured and validated in P2. KCM session policies set in P6.

Permission Model — 22 CyberArk Permissions → 4 Keeper Boolean Axes (LOSSY)

Keeper uses 4 boolean axes: manage_users, manage_records, can_edit, can_share. 12 CyberArk permissions have no Keeper equivalent and are permanently lost. Agent 03 produces a full escalation + loss report. Enterprise Security Architect must independently review every escalation flag at gate g3 — iOPEX cannot self-approve (PCI-DSS Req 7.1).

What Is This Control Center?

Executive summary: The SHIFT Migration Control Center is a real-time operational dashboard for managing the CyberArk PAS → KeeperPAM migration. It surfaces every phase, agent, gate, and risk factor across the 80-week engagement. KeeperPAM is the selected target — zero-knowledge SaaS, FedRAMP High, KSM SDK replaces CCP/AAM, KCM provides browser-native session recording.

Technical overview: The frontend is a single-page app. All data flows through the 15-agent AI orchestrator (SHIFT). Each page visualizes a different slice of the orchestrator's output — from high-level phase timelines down to individual account-level ETL results. This version is a standalone deliverable with all data inlined.

SHIFT Architecture — Framework-Agnostic Design

SHIFT is a methodology, tooling, and agent design pattern — not an SDK. The orchestration engine is swappable (LangGraph, CrewAI, AutoGen, plain Python). Agent logic, phase gates, ETL pipeline spec, and ML feedback loops carry over unchanged regardless of engine choice.
