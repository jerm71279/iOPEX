01 What Is SHIFT

SHIFT (Secure Hybrid Identity & Function Transfer) is iOPEX's PAM migration methodology, tooling, and agent design pattern. It is framework-agnostic — the orchestration engine is swappable (LangGraph, CrewAI, AutoGen, plain Python). The migration methodology, 15 agent contracts, phase gates, ETL pipeline, and ML feedback loops carry over unchanged regardless of engine choice.

Selected Target: KeeperPAM
KeeperPAM is the confirmed live migration target. Zero-knowledge SaaS architecture, FedRAMP High authorized (March 2026). Keeper Secrets Manager (KSM) replaces CyberArk CCP/AAM for application credential retrieval. Keeper Connection Manager (KCM) provides browser-native session recording.

The SHIFT Control Center is a real-time operational dashboard surfacing every phase, agent, gate, wave, and risk factor across the 80-week migration engagement. In production the backend connects to live CyberArk PVWA and KeeperPAM APIs.

02 KeeperPAM — 3-Tier Hierarchy (Structural Delta from CyberArk)

KeeperPAM uses a 3-tier object model. A single CyberArk Account maps to a pamUser record linked to a pamMachine or pamDatabase resource. The cyberark-import tool handles bulk import (~20% of work); the remaining 80% is post-import transformation to build the correct 3-tier linkage.

Each CyberArk Safe may become 2 Keeper Shared Folders — one for credentials, one for PAM resources.

PAM Config
Top-level configuration object — defines the PAM environment, gateway assignment, rotation policies, and compliance scope. Maps to the CyberArk Safe's policy settings.
▼
PAM Resource
(pamMachine / pamDatabase)
The target system — a server, database, or network device. Maps from CyberArk's address + platformId fields. One Resource may have multiple Users.
▼
PAM User
(pamUser)
The privileged credential — maps from CyberArk's userName + secret. This is the atomic unit of the ETL IMPORT step. Retrieved password is written here and discarded from memory immediately after.

05 KeeperPAM — Key Technical Details
ENCRYPTION

Zero-knowledge AES-256-GCM. Encryption/decryption is client-side — the Keeper server never sees plaintext. A full server compromise exposes only ciphertext.

COMPLIANCE

FedRAMP High authorized (March 2026). SOC 2 Type II. Supports PCI-DSS Req 7.1 dual-control gate workflow at g3 and g9.

KEEPER GATEWAY

Docker container (min 4 CPU / 16 GB RAM). Handles rotation for ~25–30 platforms natively. Outbound HTTPS to Keeper cloud only — no inbound firewall rules required. Network gear requires custom rotation scripts — validate before Wave 3.

KSM — REPLACES CCP/AAM

Keeper Secrets Manager: 40+ native integrations — GitHub Actions, Terraform, Ansible, Kubernetes, Jenkins. Agent 06 scans all app code for CCP/AAM patterns and generates KSM SDK replacements. Full re-architecture for any app using AAM agent injection.

Permission Model — 22 CyberArk Permissions → Keeper Role Model (LOSSY)
Keeper uses a vault role model — not a 1:1 map with CyberArk's 22 permissions. Agent 03 flags two critical risks: permission loss (9 permissions have no Keeper equivalent) and escalation (some members may receive more access than intended due to role rounding). Agent 03 produces a full report. iOPEX cannot self-approve this output — the Enterprise Security Architect must independently review every escalation flag at gate g3 (PCI-DSS Req 7.1 dual-control).

06 Security Architecture Q&A — KeeperPAM Focus
How does the orchestrator authenticate to KeeperPAM?
OAuth2 client credentials grant via Keeper PAM API. Client ID and secret are environment variables — never in config files or code. Token is refreshed proactively before expiry (within 60 seconds) to prevent silent token failures during a live migration run.
What happens to credentials in-flight during ETL?
All extraction uses CyberArk PVWA REST API over HTTPS (TLS enforced). Passwords retrieved via POST /Accounts/{id}/Password/Retrieve are held in memory only during the EXPORT→IMPORT step. The orchestrator zeroes password variables immediately after use. Credentials never touch disk, log files, or state files. Error sanitization strips secrets before writing to the audit log.
What happens to CyberArk audit history when we migrate?
This is the most critical compliance risk. CyberArk audit history does NOT migrate to KeeperPAM — the audit trail restarts at the migration date. Agent 07 automatically exports and archives all CyberArk audit logs before decommission in P7. Archived logs must be retained per the Enterprise compliance retention policy (7 years for SOX, 1 year for PCI-DSS minimum).
What network exposure does the Keeper Gateway require?
The Gateway runs as a Docker container and requires line-of-sight to all managed target systems inside the Enterprise perimeter. It does NOT require inbound internet exposure. Outbound HTTPS to Keeper cloud for command polling only. No inbound firewall rules needed for the Gateway itself.
How are DevOps secrets and NHIs handled?
NHI detection uses three signals: (1) platform type, (2) naming patterns (svc-, app-), (3) Safe name patterns (appcred, cicd). NHIs get a separate migration track. Keeper Secrets Manager (KSM) replaces CCP/AAM — 40+ native integrations for GitHub Actions, Terraform, Ansible, Kubernetes, Jenkins. App teams update SDK calls; Agent 06 generates the replacement code.
What is the highest-authority gate?
Gate g13 — Cutover Approval. Sets the source CyberArk vault to read-only — the last fully reversible action before decommission. All three Enterprise approvers (CAB, Exec Sponsor, Compliance) must independently authorize. If any approver withholds approval, the migration pauses indefinitely. iOPEX cannot proceed on partial or verbal authorization.
